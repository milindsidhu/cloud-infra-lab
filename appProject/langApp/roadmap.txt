🛣️ Roadmap: Interactive Script Execution in Kubernetes


🔹 Phase 1: Setup Foundation

✅ Goals

    Containerized backend app (API)

    Kubernetes cluster (with RBAC + ingress)

    Use Docker or Kaniko for image builds

🔧 Tech Stack

    Backend: Python (FastAPI), Go, or Node.js

    Kubernetes: Minikube, KinD, or Cloud (e.g., GKE, EKS)

    Storage: EmptyDir or PVC

    CI: Optional GitHub Actions for image builds

📌 Tasks

    Set up FastAPI/Express app

    Deploy it to Kubernetes (use Helm or plain YAML)

    Configure ingress (e.g., NGINX + cert-manager)

    Set up a namespace like code-exec



🔹 Phase 2: Language-Specific Job Runner

✅ Goals

    Accept code and language via API

    Spin up a pod per job (based on language)

    Stream logs from pod to frontend

🔧 Tech Stack

    Kubernetes Job API (client-python or client-go)

    Use images: python:3.12, node:20, openjdk:17, etc.

    Volume for passing code (e.g., emptyDir, configMap)

📌 Tasks

    API endpoint: POST /run

    Create pod/job YAML dynamically

    Mount code into pod

    Use kubectl logs -f or API streaming for logs

    Return Job ID or Pod name to frontend



🔹 Phase 3: Interactive Shell per Pod

✅ Goals

    Attach to shell in container from frontend

    Support input/output in real-time

🔧 Tech Stack

    WebSockets (socket.io, FastAPI WebSockets)

    Kubernetes exec + websocket bridge

    Frontend: xterm.js terminal component

📌 Tasks

    WebSocket endpoint: /shell/<pod-name>

    Use stream.stream() from kubernetes-client to attach

    Pipe WebSocket ↔ stdin/stdout of pod shell

    Handle disconnect/reconnect safely



🔹 Phase 4: Frontend (UX & Control)

✅ Goals

    UI for uploading code, selecting language

    Live output stream + shell interaction

    Optional: Save jobs and outputs

🔧 Tech Stack

    React or Next.js + xterm.js

    API: /run, /logs, /shell

    WebSocket streaming

📌 Tasks

    Upload or paste code editor

    Button to run job → call /run

    Show logs in real time via /logs

    Terminal UI using xterm.js + /shell



🔹 Phase 5: Security, Limits, Persistence

✅ Goals

    Secure sandboxing (no root access)

    Add resource limits, network policies

    Optional: Save session history

🔧 Tech Stack

    PodSecurityContext (no root)

    Resource limits (CPU/mem)

    Pod Disruption Policies

    PVC for persistent workspace

📌 Tasks

    Add runAsUser, readOnlyRootFilesystem

    Apply CPU/mem limits in pod templates

    Use NetworkPolicy to restrict traffic

    Persist user sessions in DB or volume



📦 Bonus: Cluster Requirements

    ✅ Ingress (NGINX) with TLS (cert-manager)

    ✅ Metrics server (if autoscaling)

    ✅ RBAC for job creation & pod exec

    ✅ Optional: Argo Workflows for job orchestration
